# 将上传功能集成到 Ash Repo 的实施方案

## 问题分析

### 当前架构状况
1. **Ash 基础设施已就绪**：
   - Ash Domain (`Vmemo.Photos`) 已创建
   - Photo、Note、PhotoNote Resources 已定义
   - 数据库表结构已迁移完成
   - Oban Workers 已实现同步逻辑

2. **当前上传流程**：
   - 直接调用 `TsPhoto.create()` 存储到 Typesense
   - 文件存储到本地文件系统
   - 缺少 PostgreSQL 持久化

3. **存在的问题**：
   - 上传功能仍在使用旧的 Typesense 直接存储方式
   - 没有利用已建立的 Ash Resources
   - 数据一致性和可靠性不足

## 方案对比

### 方案 A：渐进式迁移（推荐）
**优势**：
- 风险较低，可以逐步验证
- 保持现有功能稳定
- 便于调试和回滚

**实现步骤**：
1. 修改上传组件使用 Ash API
2. 保持 Typesense 同步机制
3. 逐步迁移其他功能

### 方案 B：完全重构
**优势**：
- 架构更清晰统一
- 一次性解决所有问题

**劣势**：
- 风险较高
- 可能影响现有功能
- 测试工作量大

## 技术选型

### 选择方案 A：渐进式迁移

**理由**：
1. 当前 Ash 基础设施已完备
2. 只需要修改上传流程，其他功能保持不变
3. 可以确保数据一致性

## 架构设计

### 新的上传数据流

```
用户上传照片
    ↓
LiveView 处理上传
    ↓
调用 Ash.create(Photo, attrs)  # 存储到 PostgreSQL
    ↓
触发 Oban job (after_action)
    ↓
后台异步同步到 Typesense
    ↓
搜索功能正常工作
```

### 关键修改点

1. **UploadForm 组件**：
   - 替换 `TsPhoto.create()` 为 `Ash.create(Photo, attrs)`
   - 保持文件存储逻辑不变
   - 保持用户界面不变

2. **SearchBox 组件**：
   - 同样替换为 Ash API
   - 保持搜索体验不变

3. **数据一致性**：
   - PostgreSQL 作为主存储
   - Typesense 作为搜索索引
   - Oban 确保最终一致性

## 风险评估

### 技术风险
1. **数据同步延迟**：Oban job 可能失败或延迟
   - **缓解**：已有重试机制（max_attempts: 3）
   - **监控**：Oban Web UI 监控任务状态

2. **性能影响**：额外的数据库写入
   - **评估**：影响较小，PostgreSQL 性能良好
   - **优化**：可考虑批量操作

3. **数据不一致**：PostgreSQL 和 Typesense 数据不同步
   - **缓解**：Oban job 失败重试
   - **恢复**：可重新同步数据

### 业务风险
1. **用户体验**：上传速度可能略有影响
   - **评估**：影响很小，用户感知不明显
   - **优化**：异步同步不影响响应速度

2. **功能回归**：现有功能可能受影响
   - **缓解**：保持 API 接口不变
   - **测试**：充分测试上传和搜索功能

## 实施计划

### 第一阶段：修改上传组件
1. 修改 `UploadForm` 使用 Ash API
2. 修改 `SearchBox` 使用 Ash API
3. 测试上传功能

### 第二阶段：验证和优化
1. 监控 Oban job 执行情况
2. 验证数据同步正确性
3. 性能测试和优化

### 第三阶段：清理和文档
1. 移除旧的直接 Typesense 调用
2. 更新相关文档
3. 代码清理

## 总结

采用渐进式迁移方案，将上传功能从直接 Typesense 存储迁移到 Ash + PostgreSQL + Oban 同步的架构。这个方案风险可控，能够保持现有功能稳定，同时提升数据一致性和可靠性。

关键成功因素：
1. 保持 API 接口不变
2. 确保 Oban job 稳定执行
3. 充分测试数据同步
4. 监控系统运行状态
